### PassGuard — Day 19: Content Security Policy, Shadow DOM Hardening & XSS Defense

> **Context:** Day 19 of the PassGuard build. The extension's content script injects a widget

## Part 1 — The Threat Model for an Injected Widget

When the PassGuard content script runs on `https://evil-or-compromised-site.com`, the host page's JavaScript has full access to `window`, `document`, and all non-Shadow DOM elements. The risks to an injected widget:

| Threat | Mechanism | Impact |
|---|---|---|
| **CSS bleed** | Host page global styles override widget styles | Widget becomes unusable / unreadable |
| **DOM clobbering** | Host page overwrites global variables (`window.fetch`, `window.crypto`) | Widget uses a tampered implementation |
| **Prototype pollution** | Malicious scripts modify `Array.prototype`, `Object.prototype` | Widget logic behaves incorrectly |
| **XSS via innerHTML** | Widget renders user data (password score text) using innerHTML | Host page injects script via crafted input |
| **Event hijacking** | Host page intercepts events on the widget's `<input>` | Keystrokes captured before the widget sees them |

## Part 2 — Shadow DOM: What It Protects and What It Doesn't

#### What Shadow DOM Does Protect

```javascript
// content/content.js
const host = document.createElement('div');
host.id = 'passguard-host';
document.body.appendChild(host);

const shadow = host.attachShadow({ mode: 'closed' });
//                                        ↑
// 'closed' mode: shadow root is NOT accessible via host.shadowRoot
// Even host page JS: host.shadowRoot === null
```

**`mode: 'closed'`** is the critical choice. `mode: 'open'` exposes `host.shadowRoot` to the host page's JavaScript, allowing it to reach inside the Shadow DOM and read or modify the widget. `mode: 'closed'` prevents this — the shadow root reference is only held by the content script's closure.

Shadow DOM protects against:
- ✅ CSS bleed (styles inside shadow don't leak out; host page styles don't leak in)
- ✅ Host page JS reaching into the widget's DOM
- ✅ Global CSS `!important` overrides (Shadow DOM has its own style cascade)

Shadow DOM does **not** protect against:
- ❌ DOM clobbering of globals (the content script still runs in the host page's window context)
- ❌ Prototype pollution (same window, same `Object.prototype`)
- ❌ `window.crypto` or `window.fetch` being replaced before the content script runs

## Part 3 — Fix: Caching Trusted References Before the Host Page Loads

The fix for DOM clobbering and prototype pollution: **capture trusted references to browser built-ins at content script injection time**, before any host page code can modify them.

```javascript
// content/content.js — top of file, before any host interaction

// Cache crypto before host page can replace window.crypto
const _crypto = window.crypto;
const _getRandomValues = _crypto.getRandomValues.bind(_crypto);
const _subtle = _crypto.subtle;

// Cache fetch before host page can intercept it
const _fetch = window.fetch.bind(window);

// Cache prototype methods before pollution
const _arrayPush    = Array.prototype.push;
const _objectKeys   = Object.keys.bind(Object);
const _hasOwn       = Object.prototype.hasOwnProperty;

// All PassGuard code uses _crypto, _fetch, _arrayPush etc. — never the bare globals
```

Content scripts run at `document_start` (configured in `manifest.json`) — before the host page's `<script>` tags execute. Capturing references at this point guarantees we hold the browser's authentic implementations.

```json
// manifest.json
{
  "content_scripts": [{
    "matches": ["<all_urls>"],
    "js": ["content/content.js"],
    "run_at": "document_start"  // ← critical: runs before host page scripts
  }]
}
```

## Part 4 — XSS in the Widget: innerHTML Audit

Every place the widget renders dynamic content was audited for `innerHTML` usage:

```javascript
// UNSAFE — never do this
scoreLabel.innerHTML = `Score: ${score} — ${category}`;
// If `category` contained "<script>alert(1)</script>", it would execute.

// SAFE — textContent never interprets HTML
scoreLabel.textContent = `Score: ${score} — ${category}`;

// SAFE — for structured content, build the DOM programmatically
const span = document.createElement('span');
span.className = 'score-value';
span.textContent = score.toString();
scoreLabel.appendChild(span);
```

The audit found **three** places in the original widget code that used `innerHTML` to insert score text and suggestion strings. All three were replaced with `textContent` or `createElement` / `appendChild` chains.

The only remaining `innerHTML` usage is for the static skeleton HTML injected once on widget creation (no user data involved):

```javascript
// This is safe: completely static, no dynamic data
shadow.innerHTML = `
  <div class="passguard-widget">
    <div class="score-ring-container"></div>
    <div class="score-label"></div>
    <div class="suggestions-list"></div>
  </div>
`;
```

## Part 5 — Content Security Policy for Extension Pages

Extension pages (popup, options page) support a CSP defined in `manifest.json`. The strictest valid CSP for an extension that uses no inline scripts and no external resources:

```json
// manifest.json
{
  "content_security_policy": {
    "extension_pages": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; connect-src https://api.pwnedpasswords.com; img-src 'self' data:; object-src 'none'"
  }
}
```

Breaking down each directive:

| Directive | Value | Reason |
|---|---|---|
| `default-src` | `'self'` | All resources must be bundled with the extension |
| `script-src` | `'self'` | No inline scripts, no external JS |
| `style-src` | `'self' 'unsafe-inline'` | Inline styles needed for Shadow DOM injection |
| `connect-src` | `https://api.pwnedpasswords.com` | Only permitted external request — HIBP breach check |
| `img-src` | `'self' data:` | Extension icons + any base64 SVGs |
| `object-src` | `'none'` | Block `<object>`, `<embed>`, `<applet>` entirely |

`'unsafe-inline'` in `style-src` is the one concession — Shadow DOM injection requires setting `element.style` properties dynamically. In the extension popup pages this can be removed entirely, but the content script's Shadow DOM widget requires it.

## What I Practiced / Built

- Threat-modelled the content script injection context: CSS bleed, DOM clobbering, prototype pollution, XSS via innerHTML, event hijacking
- Switched Shadow DOM from `mode: 'open'` to `mode: 'closed'` — host page JS can no longer reach the widget's internal DOM
- Implemented trusted-reference caching at `document_start` to protect `window.crypto`, `window.fetch`, and core prototype methods from host-page replacement
- Audited all `innerHTML` usage in the widget — replaced 3 dynamic insertions with `textContent` / `createElement` equivalents
- Defined a strict `content_security_policy` in `manifest.json` with `connect-src` limited to HIBP's API domain
- Understood the interaction between `run_at: document_start` and trusted reference capture timing

## Questions I Still Have

- Is `mode: 'closed'` Shadow DOM truly secure? I've read that determined attackers can still reach it via `getEventListeners` in DevTools — but that's a DevTools capability, not page JS. Is that a meaningful distinction?
- `'unsafe-inline'` in `style-src` feels wrong even though it's limited to styles. Is there a way to use a nonce for the Shadow DOM style injection?
- Should the content script defend against timing attacks where the host page measures how long PassGuard's analysis takes to infer the password being typed?
- How does Firefox handle `document_start` timing vs Chrome — is the guarantee identical?
