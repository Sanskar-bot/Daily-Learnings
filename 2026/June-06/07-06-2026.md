#### `argon2.ts` — Deriving the Key Encryption Key

Argon2id takes the master password and a random per-user salt and produces a deterministic 32-byte KEK. The same password + same salt always produces the same KEK — this is how decryption works across sessions and devices.

```typescript
// argon2.ts
import sodium from 'libsodium-wrappers-sumo';

export interface Argon2Params {
  readonly saltHex: string;      // 32 bytes, hex-encoded, stored in DB
  readonly opsLimit?: number;    // iterations — default 3
  readonly memLimit?: number;    // memory in bytes — default 67108864 (64 MB)
}

export async function deriveKEK(
  masterPassword: string,
  params: Argon2Params
): Promise<Uint8Array> {
  await sodium.ready;

  const salt = sodium.from_hex(params.saltHex);
  const password = sodium.from_string(masterPassword);

  return sodium.crypto_pwhash(
    32,                                      // output length: 32 bytes → KEK
    password,
    salt,
    params.opsLimit ?? 3,
    params.memLimit ?? sodium.crypto_pwhash_MEMLIMIT_INTERACTIVE,
    sodium.crypto_pwhash_ALG_ARGON2ID13
  );
}

export function generateArgon2Salt(): string {
  const salt = crypto.getRandomValues(new Uint8Array(32));
  return Buffer.from(salt).toString('hex');
}
```

Key decisions:
- **`crypto.getRandomValues()`** — never `Math.random()`. `crypto.getRandomValues()` uses the OS CSPRNG (Cryptographically Secure Pseudo-Random Number Generator).
- **`sodium.ready` awaited** — libsodium is a WebAssembly module; the Wasm binary must be compiled before any function call. Every function that calls libsodium must `await sodium.ready` first.
- **`MEMLIMIT_INTERACTIVE`** — 64 MB. High enough to make GPU/ASIC attacks expensive, low enough to run on mobile.

#### `aes-gcm.ts` — Authenticated Encryption

AES-256-GCM is the workhorse. Every piece of data that leaves the device goes through this function. The `encrypt` function:

1. Generates a **fresh random 32-byte IV** (nonce) every call — reusing an IV with the same key is a catastrophic GCM failure
2. Encrypts + authenticates in one pass, producing ciphertext + a 16-byte authentication tag
3. **Prepends the IV** to the output blob so the decryptor always has it

```typescript
// aes-gcm.ts
export async function encryptAESGCM(
  plaintext: Uint8Array,
  key: Uint8Array
): Promise<Uint8Array> {
  try {
    const iv = crypto.getRandomValues(new Uint8Array(12)); // 96-bit IV for GCM
    const cryptoKey = await crypto.subtle.importKey(
      'raw', key, { name: 'AES-GCM' }, false, ['encrypt']
    );
    const ciphertext = await crypto.subtle.encrypt(
      { name: 'AES-GCM', iv },
      cryptoKey,
      plaintext
    );
    // Output: [12-byte IV][ciphertext + 16-byte authTag]
    const result = new Uint8Array(12 + ciphertext.byteLength);
    result.set(iv, 0);
    result.set(new Uint8Array(ciphertext), 12);
    return result;
  } catch (err) {
    throw new Error(`AES-GCM encryption failed: ${String(err)}`);
  }
}

export async function decryptAESGCM(
  blob: Uint8Array,
  key: Uint8Array
): Promise<Uint8Array> {
  try {
    if (blob.length < 28) throw new Error('Blob too short — corrupted or invalid');
    const iv         = blob.slice(0, 12);
    const ciphertext = blob.slice(12);  // includes trailing 16-byte authTag
    const cryptoKey  = await crypto.subtle.importKey(
      'raw', key, { name: 'AES-GCM' }, false, ['decrypt']
    );
    const plaintext = await crypto.subtle.decrypt(
      { name: 'AES-GCM', iv },
      cryptoKey,
      ciphertext
    );
    return new Uint8Array(plaintext);
  } catch (err) {
    // Deliberately vague — do not reveal whether the key or IV was wrong
    throw new Error('Decryption failed — wrong key or corrupted data');
  }
}
```

Why the error message is deliberately vague in `decryptAESGCM`: an attacker who can trigger decryption and observe different errors for "wrong key" vs "corrupted data" has an oracle — a side channel. The message "wrong key or corrupted data" collapses both cases intentionally.

#### `keys.ts` — Vault Key Wrap & Unwrap

The vault key is a random 32-byte key that encrypts the actual vault entries. It is generated once at registration and stored *wrapped* (encrypted) by the KEK.

```typescript
// keys.ts
export function generateVaultKey(): Uint8Array {
  return crypto.getRandomValues(new Uint8Array(32));
}

// Wrap: encrypt vault key with KEK, produce base64 blob for server storage
export async function wrapVaultKey(
  vaultKey: Uint8Array,
  kek: Uint8Array
): Promise<string> {
  const wrapped = await encryptAESGCM(vaultKey, kek);
  return Buffer.from(wrapped).toString('base64');
}

// Unwrap: decrypt vault key from base64 blob using KEK
export async function unwrapVaultKey(
  wrappedBase64: string,
  kek: Uint8Array
): Promise<Uint8Array> {
  const wrapped = Buffer.from(wrappedBase64, 'base64');
  return decryptAESGCM(new Uint8Array(wrapped), kek);
}
```

This function is called exactly twice per user session:
1. **On login** — `unwrapVaultKey(user.wrappedVaultKey, kek)` → vault key held in service worker memory
2. **On master password change** — `wrapVaultKey(vaultKey, newKEK)` → new wrapped blob stored in DB

#### Why WebCrypto (`crypto.subtle`) not libsodium for AES-GCM?

libsodium is used for Argon2id because the Web Crypto API (`crypto.subtle`) has no Argon2 implementation. But for AES-256-GCM, the Web Crypto API is preferred:

| | WebCrypto (`crypto.subtle`) | libsodium-wrappers |
|--|--|--|
| AES-256-GCM | ✅ Native browser support | ✅ Available |
| Performance | ✅ Hardware-accelerated (AES-NI) | Wasm — slower |
| Bundle size impact | ✅ Zero — built-in | Adds ~300KB Wasm |
| Availability | Chrome, Firefox, Safari, Node 15+ | Any environment |

Decision: use `crypto.subtle` for AES-GCM (faster, zero bundle cost), libsodium for Argon2id (no alternative).
