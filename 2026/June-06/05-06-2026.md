#### Why Zero-Knowledge? The Threat Model First

Before choosing any tool or library, I defined the threat model. A password manager is only as trustworthy as what it *cannot* do — so I started with constraints, not features.

| Threat | Mitigation |
|--------|-----------|
| Server breach | Ciphertext-only storage — server never holds plaintext or keys |
| Insider threat | No server-side decryption path exists — architecture enforces this |
| Database dump | Encrypted blobs are useless without the user's master password |
| Phishing attack | Levenshtein hostname comparison before any autofill occurs |
| Token theft | Refresh tokens are single-use, rotated on every call, stored as SHA-256 hashes |

The non-negotiable rule: **the master password never leaves the device.** Every decision that followed was constrained by this.

#### Key Derivation — Why Argon2id over bcrypt/scrypt

| Algorithm | Memory-Hard | GPU Resistant | Side-Channel Safe | Recommended |
|-----------|-------------|---------------|-------------------|-------------|
| bcrypt | ❌ | Partially | ❌ | Legacy |
| scrypt | ✅ | ✅ | ❌ | Older |
| **Argon2id** | ✅ | ✅ | ✅ | **✅ NIST / RFC 9106** |

Argon2id is the winner of the Password Hashing Competition (2015) and is the current NIST recommendation. The `id` variant combines the data-dependent memory access of Argon2d (GPU resistance) with the data-independent passes of Argon2i (side-channel resistance) — the best of both.

Parameters chosen for VaultZero:
- **Memory:** 64 MB (`65536 KiB`) — expensive for attackers to parallelise
- **Iterations:** 3 — sequential, cannot be shortcut
- **Parallelism:** 1 — single-threaded derivation (safe for extension/mobile)
- **Output:** 32 bytes → the KEK (Key Encryption Key)

#### Encryption — Why AES-256-GCM

AES-256-GCM provides **authenticated encryption** — it encrypts *and* authenticates in a single pass. The authentication tag (`authTag`) detects any tampering or corruption before decryption even begins. This is critical: a password manager must detect bit-flipping attacks on the stored blob, not silently decrypt garbage.

```
encrypt(plaintext, key):
  iv  = crypto.getRandomValues(32 bytes)   ← fresh random IV every call
  {ciphertext, authTag} = AES-256-GCM(key, iv, plaintext)
  return iv + ciphertext + authTag          ← IV prepended, always

decrypt(blob, key):
  iv         = blob[0..31]
  ciphertext = blob[32..n-16]
  authTag    = blob[n-16..n]
  return AES-256-GCM.decrypt(key, iv, ciphertext, authTag)
```

The IV is never reused — GCM security collapses if the same (key, IV) pair encrypts two different messages.

#### Two-Layer Key Architecture

Encrypting the vault directly with the Argon2id-derived key has a fatal flaw: changing the master password requires re-encrypting the entire vault. The two-layer design solves this elegantly:

```
Master Password + Salt
  └─→ Argon2id → KEK (Key Encryption Key, 32 bytes)
        └─→ AES-256-GCM wraps → Vault Key (random, 32 bytes)
              └─→ AES-256-GCM encrypts → Vault Entries (all passwords)
```

Changing the master password = derive new KEK, re-wrap vault key = **one tiny operation**, zero vault re-encryption.

#### Monorepo Design — npm Workspaces

The project spans 4 platforms (extension, web, Android, iOS) and a backend. Sharing the `@vaultzero/core` crypto library across all JavaScript/TypeScript surfaces requires a monorepo.

Evaluated options:

| Tool | Complexity | npm Workspaces Support | Rationale |
|------|-----------|----------------------|-----------|
| Turborepo | Medium | ✅ | Overkill for Day 1 |
| Nx | High | ✅ | Too opinionated |
| **npm Workspaces** | Low | ✅ | Sufficient, zero config |
| Lerna | Medium | ✅ | Deprecated in favour of workspaces |

Decision: **npm workspaces** — simple, built-in, no additional tooling required at this stage.

#### Database Design — Key Decisions

- **`argon2Salt`** stored per-user — unique salt makes every KEK unique even if two users share the same master password
- **`wrappedVaultKey`** stored as base64 — the vault key encrypted with the user's KEK, never the raw vault key
- **`RefreshToken.tokenHash`** stores SHA-256 of the raw token — the raw token is never persisted, only its hash
- **`RefreshToken.used`** boolean — single-use rotation, prevents replay attacks on stolen tokens
- **`AuditLog`** — every vault access, login, and password change is logged with IP and device — non-repudiation
