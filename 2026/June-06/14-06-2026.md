## Part 1 — Kali Linux Dual Boot Installation

#### Why Dual Boot Over a VM?

Virtual machines (VirtualBox, VMware) are convenient but come with real limitations for security work:

- **Performance** — A VM shares CPU and RAM with the host. Tools like `hashcat` (GPU-based password cracking) can't properly access the GPU in a VM — Kali on bare metal has direct GPU access.
- **Networking** — VM NAT/bridged adapters add a layer that can interfere with packet capture (Wireshark, `tcpdump`) and wireless injection attacks. Native adapters behave exactly as expected.
- **USB device passthrough** — Rubber Ducky, WiFi adapters in monitor mode, and hardware security keys often don't pass through cleanly to VMs.

Dual boot gives a native Kali environment that shares disk space with Windows, with a bootloader letting you choose OS on startup.

#### Pre-Installation Checklist

Before touching the installer:

1. **Backed up Windows** — created a system restore point and backed up important files. Partitioning errors can wipe a drive; a backup is non-negotiable.
2. **Disabled BitLocker** — BitLocker encrypts the Windows partition. If the bootloader is modified (which dual-boot does), Windows can fail to boot and demand a recovery key. Disabled before proceeding.
3. **Disabled Secure Boot** — Kali's bootloader (GRUB) may not be signed with Microsoft's keys on older setups. Disabled Secure Boot in UEFI/BIOS settings.
4. **Disabled Fast Startup** — Windows Fast Startup writes a hibernation snapshot on shutdown. If Kali mounts the Windows NTFS partition while this snapshot exists, it can corrupt the Windows partition. Disabled in Power Options → Choose what the power buttons do.
5. **Downloaded Kali ISO** — from `kali.org/get-kali` — the official "Installer" image (not Live). Verified the SHA256 hash against the published checksum to ensure the ISO wasn't tampered with.
6. **Created bootable USB** — used **Rufus** on Windows with GPT partition scheme (matching the system's UEFI mode) and DD write mode for the Kali ISO.

#### Partitioning

Shrunk the Windows C: drive by **80 GB** using Windows Disk Management (not a third-party tool — safer for NTFS). This created 80 GB of unallocated space that Kali's installer would use.

During Kali installation, chose **Manual** partitioning:

| Partition | Size | Type | Mount Point | Purpose |
|---|---|---|---|---|
| EFI System Partition | (shared with Windows) | FAT32 | `/boot/efi` | GRUB bootloader |
| Root | 70 GB | ext4 | `/` | Kali OS + tools |
| Swap | 8 GB | swap | — | RAM overflow / hibernate |

The EFI System Partition already exists on UEFI systems — Kali's installer simply installs GRUB into it alongside Windows Boot Manager.

#### GRUB Bootloader

After installation, GRUB replaced Windows Boot Manager as the first-stage bootloader. On startup, GRUB presents a menu:

```
┌─────────────────────────────────────────┐
│  Kali GNU/Linux                         │  ← default (highlighted)
│  Kali GNU/Linux (recovery mode)         │
│  Windows Boot Manager (on /dev/nvme0n1) │
│                                         │
└─────────────────────────────────────────┘
```

Windows is still fully accessible — just select it from the GRUB menu. Both OSes are completely independent on their own partitions.

#### Post-Install Setup

After booting into Kali:

```bash
sudo apt update && sudo apt full-upgrade -y   # update all packages
sudo apt install -y kali-linux-everything     # full Kali metapackage — all tools
```

`kali-linux-everything` installs the complete Kali tool suite — Metasploit, Burp Suite, Wireshark, Nmap, Aircrack-ng, John the Ripper, Hashcat, and ~600 more tools. Takes ~15 minutes on a decent connection.

## Part 2 — GoPhish: First Look

#### What Is GoPhish?

**GoPhish** is an open-source phishing simulation framework written in Go. It's used by:
- **Red teams** — to simulate phishing attacks during authorized penetration tests
- **Security awareness teams** — to run internal phishing drills and measure how many employees click malicious links or submit credentials
- **Researchers** — to study phishing infrastructure

It provides a full phishing campaign stack in a single binary: SMTP sender, HTTP landing page server, campaign tracking, and a web dashboard.

#### Installation on Kali

```bash
# Download the latest release binary
wget https://github.com/gophish/gophish/releases/download/v0.12.1/gophish-v0.12.1-linux-64bit.zip

# Unzip
unzip gophish-v0.12.1-linux-64bit.zip -d gophish/
cd gophish/

# Make executable and run
chmod +x gophish
sudo ./gophish
```

On first run, GoPhish generates a random admin password and prints it to stdout:

```
time="2026-06-14T..." level=info msg="Please login with the username admin and the password <RANDOM_PASSWORD>"
```

The admin dashboard runs on `https://localhost:3333` (self-signed cert — browser will warn, that's expected).

#### Dashboard Overview

The GoPhish dashboard has 5 core concepts:

| Component | What It Does |
|---|---|
| **Users & Groups** | Target list — names, emails of people to be phished |
| **Email Templates** | HTML email body with tracking pixel and phishing link |
| **Landing Pages** | Cloned/custom HTML page that captures credentials |
| **Sending Profiles** | SMTP server config — the email account GoPhish sends from |
| **Campaigns** | Ties all the above together: who, what email, what landing page, when |

#### How a Campaign Works (Conceptual)

1. **Create a Group** — add target email addresses (in a real engagement, these are the users being tested).
2. **Create an Email Template** — craft a convincing phishing email. GoPhish supports HTML and provides `{{.FirstName}}`, `{{.Email}}`, `{{.TrackingURL}}` template variables.
3. **Clone a Landing Page** — GoPhish can fetch any URL and clone its HTML to create a credential-harvesting page. It automatically injects a form POST that sends submitted credentials to GoPhish's backend.
4. **Configure a Sending Profile** — SMTP credentials (host, port, username, password, `From` address).
5. **Launch the Campaign** — GoPhish sends the phishing emails at the scheduled time and tracks:
   - **Email opened** (via 1×1 tracking pixel)
   - **Link clicked** (via unique tracking URL per target)
   - **Credentials submitted** (via landing page form)
   - **Reported** (if the user reports the email as phishing)

#### What I Explored

- Navigated the full dashboard — Users & Groups, Email Templates, Landing Pages, Sending Profiles, Campaigns
- Cloned a simple login page (used a local HTML mockup — no external sites) to see how the landing page injection works
- Looked at the tracking URL format: `http://<phishing-domain>/<rid>` where `rid` is a unique per-target identifier
- Inspected the API — GoPhish exposes a full REST API on port `3333`, usable with an API key for automation
- Read through `config.json` — GoPhish's config file controls ports, TLS certificates, admin credentials, and the phishing server domain

#### Key Security & Ethics Notes

- GoPhish is a **dual-use tool** — it's legitimate for authorized security testing and completely illegal if used against real people without authorization.
- All exploration today was **local only** — no emails sent, no external infrastructure, no real targets.
- In a real red team engagement, GoPhish is always deployed under a domain the team controls, with proper SSL certificates (Let's Encrypt), and only against explicitly authorized targets defined in the Rules of Engagement (RoE).
