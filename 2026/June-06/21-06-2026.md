## Part 1 — Browser Extension Architecture

#### Manifest V3 Anatomy

A Chrome/Edge extension is driven by `manifest.json` — the contract between the extension and the browser. The three key execution contexts are:

| Context | File | Lifetime | DOM Access |
|---|---|---|---|
| **Popup** | `popup.html` / React app | Only while popup is open | ✅ Own popup DOM |
| **Background** | `service_worker.js` | Event-driven, short-lived | ❌ No DOM |
| **Content Script** | `content.js` | Lives on the host page | ✅ Host page DOM |

Key insight: these three contexts **cannot share memory** — they communicate exclusively via `chrome.runtime.sendMessage` / `chrome.runtime.onMessage` and `chrome.storage`.

#### Message Passing Pattern

```javascript
// popup.js — send a request to background
chrome.runtime.sendMessage(
  { action: 'getPasswords', siteUrl: window.location.href },
  (response) => {
    if (response.success) renderPasswords(response.data);
  }
);

// background service_worker.js — handle the request
chrome.runtime.onMessage.addListener((msg, sender, sendResponse) => {
  if (msg.action === 'getPasswords') {
    getStoredPasswords(msg.siteUrl).then(data => {
      sendResponse({ success: true, data });
    });
    return true; // keeps the message channel open for async response
  }
});
```

The `return true` in the background listener is critical — without it, the message channel closes before the async `getStoredPasswords` promise resolves and `sendResponse` becomes a no-op.

#### Storage: `chrome.storage.local` vs `chrome.storage.sync`

| | `local` | `sync` |
|---|---|---|
| **Quota** | 10 MB | 100 KB |
| **Synced across devices** | ❌ | ✅ (via Google account) |
| **Use case** | Large encrypted vaults | Small settings/preferences |

PassGuard uses `chrome.storage.local` for the encrypted password vault (can be large) and `chrome.storage.sync` for user preferences like theme and auto-lock timeout.

## Part 2 — PassGuard Extension UI

#### App Shell Structure

The extension popup renders a full React application at `extension.html`. The shell is split into:

- **Auth layer** — master password entry, biometric unlock prompt
- **Vault view** — searchable list of saved credentials grouped by domain
- **Generator** — password strength calculator and random password generator
- **Settings** — auto-lock timer, theme, export/import vault

#### Password Strength Algorithm

Implemented a real-time strength scorer using entropy estimation:

```javascript
function calcEntropy(password) {
  const charsets = [
    { regex: /[a-z]/, size: 26 },
    { regex: /[A-Z]/, size: 26 },
    { regex: /[0-9]/, size: 10 },
    { regex: /[^a-zA-Z0-9]/, size: 32 },
  ];

  const poolSize = charsets.reduce(
    (acc, { regex, size }) => acc + (regex.test(password) ? size : 0),
    0
  );

  return Math.log2(Math.pow(poolSize, password.length));
}

// Entropy → strength label
function strengthLabel(bits) {
  if (bits < 28) return { label: 'Very Weak', color: '#ef4444' };
  if (bits < 36) return { label: 'Weak',      color: '#f97316' };
  if (bits < 60) return { label: 'Fair',       color: '#eab308' };
  if (bits < 80) return { label: 'Strong',     color: '#22c55e' };
  return              { label: 'Very Strong', color: '#10b981' };
}
```

Entropy formula: `log₂(poolSize^length)` = `length × log₂(poolSize)`. A 12-character password using all four character classes has a pool of 94, giving ~78 bits — solidly "Strong".

#### Auto-Lock Implementation

The extension locks after a configurable idle period. The tricky part: the background service worker is event-driven and doesn't run continuously, so `setTimeout` can't be used for auto-lock.

Solution: store the `lockAt` timestamp in `chrome.storage.local` and check it on every popup open:

```javascript
// On popup open
async function checkAutoLock() {
  const { lockAt } = await chrome.storage.local.get('lockAt');
  if (lockAt && Date.now() > lockAt) {
    await lockVault();
  }
}

// On successful unlock — set new lockAt
async function onUnlock(timeoutMinutes) {
  const lockAt = Date.now() + timeoutMinutes * 60 * 1000;
  await chrome.storage.local.set({ lockAt });
}
```

This is robust against service worker restarts — the state lives in storage, not in process memory.
