### VaultZero — Railway Deployment: Debugging & Going Live

> **Context:** VaultZero's backend is a TypeScript monorepo (`@vaultzero/backend`) built on
> Fastify + Prisma + PostgreSQL. Today's goal: deploy it to **Railway** — a modern PaaS platform
> that handles infrastructure (PostgreSQL, environment variables, auto-deploys from GitHub).
> The deployment was not smooth — several issues surfaced one after another. Each was diagnosed
> and resolved with AI assistance. By end of day, the backend was live on Railway.

---

#### What Is Railway?

Railway is a deployment platform similar in spirit to Heroku but built for modern stacks. It provisions infrastructure (databases, Redis, etc.) as first-class services alongside your app, injects environment variables automatically, and deploys from a GitHub push. For a Fastify + Prisma + PostgreSQL backend, it's a natural fit: Railway provisions a managed PostgreSQL instance and wires `DATABASE_URL` into the app's environment automatically.

#### Issue 1 — Build Failing: `Cannot find module '@vaultzero/core'`

**What happened:** Railway cloned the repo and ran `npm install && npm run build`. The build failed immediately:

```
error TS2307: Cannot find module '@vaultzero/core' or its corresponding type declarations.
```

**Root cause:** VaultZero is a monorepo using npm workspaces. `@vaultzero/core` is a local workspace package. Railway was only installing dependencies for the root, not running the workspace build — so `@vaultzero/core` was never compiled to `dist/`, meaning its TypeScript declarations didn't exist when `@vaultzero/backend` tried to import them.

**Fix (AI-assisted):** Added a `nixpacks.toml` config file at the repo root to tell Railway exactly how to build the monorepo:

```toml
# nixpacks.toml
[phases.setup]
nixPkgs = ["nodejs_20"]

[phases.install]
cmds = ["npm install"]

[phases.build]
cmds = [
  "npm run build --workspace=packages/core",
  "npm run build --workspace=packages/backend"
]

[start]
cmd = "node packages/backend/dist/server.js"
```

Building `core` before `backend` ensures the declarations exist when the backend compiler runs.

#### Issue 2 — `prisma generate` Not Running Before the Server Starts

**What happened:** Even after fixing the build order, the server crashed on boot:

```
PrismaClientInitializationError: Prisma Client is not generated.
Run `prisma generate` to fix this.
```

**Root cause:** Prisma generates a type-safe client from the schema at build time (`npx prisma generate`). Railway's default build didn't run this step, so the generated client code wasn't present in `node_modules/@prisma/client`.

**Fix (AI-assisted):** Added `prisma generate` to the `postinstall` script in `packages/backend/package.json`:

```json
{
  "scripts": {
    "postinstall": "prisma generate",
    "build": "tsc --noEmit false --outDir dist",
    "start": "node dist/server.js"
  }
}
```

`postinstall` runs automatically after every `npm install` — on Railway, on every developer's machine, and in CI. Prisma's client is always generated before any build step runs.

#### Issue 3 — Database Migration Not Applied: Tables Don't Exist

**What happened:** Server started, accepted the first POST request to `/auth/register`, then crashed:

```
PrismaClientKnownRequestError: 
The table `public.User` does not exist in the current database.
```

**Root cause:** Railway provisioned a fresh PostgreSQL database — completely empty. Prisma migrations (`prisma/migrations/`) define the schema, but they only run when `prisma migrate deploy` is explicitly called. Railway never ran it.

**Fix (AI-assisted):** Added a migration step to the Railway start command. The correct pattern for production is `migrate deploy` (not `migrate dev` — `dev` is for local development only and creates shadow databases):

Updated `nixpacks.toml` start command:

```toml
[start]
cmd = "npx prisma migrate deploy && node packages/backend/dist/server.js"
```

`prisma migrate deploy` applies all pending migrations in order, is idempotent (safe to run on every deploy), and exits with a non-zero code if any migration fails — which causes Railway to mark the deploy as failed rather than silently serving a broken app.

#### Issue 4 — Port Binding: Server Not Accessible

**What happened:** Migrations ran, server started successfully (logs showed `Server listening on port 3001`), but Railway marked the service as "unhealthy" and the public URL returned a 502.

**Root cause:** Railway injects the port your app must listen on via the `PORT` environment variable. The server was hardcoded to `3001`. Railway's health check probes the `PORT`-assigned port — a mismatch means the health check fails and traffic is never routed to the container.

**Fix (AI-assisted):** Updated `packages/backend/src/server.ts` to read from `process.env.PORT`:

```typescript
// server.ts — before
const PORT = 3001;

// server.ts — after
const PORT = parseInt(process.env.PORT ?? '3001', 10);

await server.listen({ port: PORT, host: '0.0.0.0' });
//                                  ^^^^^^^^^^^
// host: '0.0.0.0' is also required — Railway routes traffic to the container's
// network interface, not just localhost. Without this, the server binds to 127.0.0.1
// and is unreachable from outside the container.
```

Two fixes in one: read `PORT` from environment, and bind to `0.0.0.0` (all interfaces) not `localhost`.

#### Issue 5 — `DATABASE_URL` Missing: Environment Variable Not Linked

**What happened:** After fixing the port, the deploy succeeded but `prisma migrate deploy` started failing on subsequent redeploys:

```
Error: Environment variable not found: DATABASE_URL.
```

**Root cause:** Railway's PostgreSQL service exposes connection credentials as environment variables, but they need to be explicitly *referenced* in the service that uses them. The backend service wasn't linked to the Postgres service — so `DATABASE_URL` wasn't being injected.

**Fix:** In Railway dashboard → Backend service → Variables tab → "Add Reference" → selected the Postgres service's `DATABASE_URL`. Railway then injects the full connection string (including host, port, credentials, database name) at runtime. No hardcoded credentials anywhere in the codebase.

---

#### Issue 6 — Argon2id Native Binary: Build Architecture Mismatch

**What happened:** After all the above fixes, the final issue: the `argon2` npm package uses a native C++ addon compiled at install time. The binary compiled on my Windows dev machine (if node_modules was committed) is not compatible with Railway's Linux (Debian) containers.

**Root cause:** `node_modules` should never be committed to git, but a stale `.gitignore` was missing `packages/*/node_modules`. Railway was picking up the Windows-compiled `argon2` binary.

**Fix (AI-assisted):**
1. Updated `.gitignore` to exclude all nested `node_modules`:
   ```
   # .gitignore
   node_modules/
   packages/*/node_modules/
   packages/*/dist/
   ```
2. Ran `git rm -r --cached packages/backend/node_modules` to untrack the committed binaries.
3. Pushed — Railway re-installed everything fresh on Linux, compiled the `argon2` native addon for the correct architecture. ✅

#### Key Lessons — Deploying a TypeScript Monorepo to Railway

| Issue | Root Cause | Fix |
|---|---|---|
| Module not found | Workspace packages not built in order | `nixpacks.toml` with explicit build sequence |
| Prisma client missing | `prisma generate` never ran | `postinstall` script in `package.json` |
| Tables don't exist | Migrations never applied | `prisma migrate deploy` in start command |
| 502 / unhealthy | Wrong port + localhost binding | `process.env.PORT` + `host: '0.0.0.0'` |
| `DATABASE_URL` missing | Service not linked in Railway dashboard | Reference Postgres service variables |
| Native binary crash | Windows binary committed, Linux needed | Fix `.gitignore`, untrack + reinstall |
