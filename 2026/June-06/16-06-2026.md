## What I Learned

### PassGuard — Day 16: Web Worker Offloading & Performance Profiling

> **Context:** Day 16 of the PassGuard build. After shipping the HIBP breach check yesterday,
> the profiler revealed a real performance problem: on long passwords (28+ characters) with
> complex patterns, the combined Trie traversal + personal dictionary generation was blocking the
> main thread for 40–80ms — enough to stutter the score ring animation. Today: move all CPU-heavy
> analysis into a dedicated Web Worker, profile the before/after, and ensure the UI thread only
> ever handles DOM updates.

---

## Part 1 — Why the Main Thread Was Blocking

#### Chrome DevTools Performance Trace — Before

Running a 30-character password through the full pipeline produced this main-thread profile:

```
Main Thread
├── onPasswordInput (debounced at 120ms)
│   ├── analyseStrength()          ~0.8ms   ✅ fine
│   ├── normalizeLeet()            ~1.2ms   ✅ fine
│   ├── checkWordlist()            ~34ms    ⚠️  Trie on 30-char input = O(N×K)
│   ├── detectPatterns()           ~3ms     ✅ fine
│   ├── checkUsername()            ~0.5ms   ✅ fine
│   └── generatePersonalDict()    ~48ms    ❌ 5,000–15,000 combinatorial strings
│
│   Total blocking time: ~87ms
│   Animation frame budget: 16ms (60fps)
│   Frames dropped: 5–6
```

The Trie substring scan (`checkWordlist`) and the personal dictionary generator (`generatePersonalDict`) together consumed 82ms of a 16ms frame budget — the score ring animation stuttered visibly on every keystroke.

#### The Fix: Web Workers

A **Web Worker** runs JavaScript in a separate OS thread, with its own call stack, heap, and event loop. It communicates with the main thread exclusively via `postMessage` / `onmessage` — no shared memory (unless using `SharedArrayBuffer`, which we don't need here). The main thread is never blocked.

## Part 2 — Implementing `analysis.worker.js`

```javascript
// workers/analysis.worker.js

import { analyseStrength }       from '../shared/strength.js';
import { detectPatterns,
         normalizeLeet }         from '../shared/patterns.js';
import { checkWordlist }         from '../shared/wordlist.js';
import { checkUsername }         from '../shared/username.js';
import { generatePersonalDict }  from '../shared/personalDictionaryGenerator.js';
import { scorePersonal }         from '../shared/personalDictionaryScorer.js';

self.onmessage = function(event) {
  const { password, username, profile, id } = event.data;

  // All CPU-heavy work happens here — off the main thread
  const normalized = normalizeLeet(password);
  const strength   = analyseStrength(password);
  const wordlist   = checkWordlist(normalized);
  const patterns   = detectPatterns(password);
  const usernameR  = checkUsername(password, username);

  let personalResult = null;
  if (profile) {
    const dict   = generatePersonalDict(profile);  // 5,000–15,000 entries
    personalResult = scorePersonal(password, normalized, dict);
  }

  // Post all results back in one message
  self.postMessage({ id, strength, wordlist, patterns, usernameR, personalResult });
};
```

Key design decisions:

- **Single worker instance** — created once on page load, reused for all messages. Creating a new Worker per keystroke would be catastrophic (Worker startup is ~5ms).
- **Message ID (`id`)** — each analysis request includes a unique ID (timestamp or counter). The main thread only applies results whose ID matches the *latest* request, discarding stale responses from previous keystrokes.
- **No `SharedArrayBuffer`** — would require COOP/COEP headers that complicate the static Vercel deployment. Pure `postMessage` serialization (structured clone) is sufficient for the data sizes involved.

#### Main Thread — `app.js` after refactor

```javascript
// app.js
const worker = new Worker('./workers/analysis.worker.js', { type: 'module' });
let latestRequestId = 0;

worker.onmessage = ({ data }) => {
  // Discard stale responses
  if (data.id !== latestRequestId) return;

  const score = computeScore(
    data.strength, data.wordlist, data.patterns,
    data.usernameR, data.personalResult, currentBreachResult
  );
  updateUI(score, data);
};

function onPasswordInput(password) {
  const id = ++latestRequestId;

  // UI responds instantly with a "computing..." state
  setLoadingState(true);

  // Offload all analysis
  worker.postMessage({ id, password, username: currentUser, profile: currentProfile });

  // Breach check runs in parallel (async, separate path)
  checkBreach(password).then(breachResult => {
    currentBreachResult = breachResult;
    // If worker has already responded, recompute score with breach data
    if (currentAnalysis) {
      const score = computeScore(...currentAnalysis, breachResult);
      updateScoreOnly(score, breachResult);
    }
  });
}
```

## Part 3 — Profiling After the Refactor

```
Main Thread (after)
├── onPasswordInput
│   ├── postMessage to worker    ~0.1ms   ✅ instant
│   ├── setLoadingState()        ~0.2ms   ✅ instant
│   └── checkBreach() (async)   ~0.0ms   ✅ non-blocking
│
│   Total blocking time: ~0.3ms
│   Animation frame budget: 16ms
│   Frames dropped: 0

Worker Thread
├── normalizeLeet()              ~1.2ms
├── analyseStrength()            ~0.8ms
├── checkWordlist()              ~34ms   (off main thread — doesn't matter)
├── detectPatterns()             ~3ms
├── generatePersonalDict()       ~48ms   (off main thread — doesn't matter)
└── postMessage result back      ~0.5ms
```

The main thread is now 0.3ms per keystroke. The score ring animates smoothly at 60fps even while the worker is doing 80ms of analysis in the background.

## Part 4 — Extension Web Worker Caveat

In Manifest V3, **content scripts cannot create Web Workers directly** — the Worker's script URL must be listed as a `web_accessible_resource`. The extension's analysis worker is already registered this way from Day 12, so the content script can spin up the worker using:

```javascript
const workerUrl = chrome.runtime.getURL('workers/dictionary.worker.js');
const worker = new Worker(workerUrl, { type: 'module' });
```

The new `analysis.worker.js` gets the same treatment — added to `web_accessible_resources` in `manifest.json`.

## What I Practiced / Built

- Profiled the main-thread blocking with Chrome DevTools Performance panel — identified Trie scan (34ms) and personal dict generation (48ms) as the two offenders
- Implemented `analysis.worker.js` — a single long-lived Web Worker that handles all CPU-bound analysis off the main thread
- Added a request-ID discard pattern to prevent stale Worker responses from overwriting newer UI state
- Refactored `app.js` to use `postMessage` / `onmessage` with a parallel async breach check

- Registered the new worker in `web_accessible_resources` for the extension content script context
