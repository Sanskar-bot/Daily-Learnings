## Part 1 — The k-Anonymity Model (Why It's Brilliant)

#### The Problem: You Can't Send a Password Over the Wire

The naive implementation of "check if this password is breached" would be to POST the plaintext password to a remote API. This is obviously catastrophic — you'd be transmitting the exact secret you're trying to protect to a third party, over a network, on every keystroke.

#### How k-Anonymity Solves It

HIBP's Pwned Passwords API uses a **range query** model:

1. Hash the password with SHA-1 locally: `SHA1("Password123") → "b2c4ef..."` (40 hex chars)
2. Send only the **first 5 characters** of the hash to the API: `GET /range/B2C4E`
3. The API returns **all hashes in its database that start with those 5 chars** — typically 400–900 suffixes and their breach counts
4. Search the returned list locally for the remaining 35 characters of your full hash
5. If found, the password is breached. The server **never saw your full hash, let alone your password**

The "k" in k-Anonymity: your query is indistinguishable from ~800 other legitimate queries that share the same 5-character prefix. The server cannot determine which specific password you checked.

```
Client                              HIBP API
  │                                    │
  │  SHA1("Password123")               │
  │  = "B2C4EF245872B2A3C5040FE..."   │
  │                                    │
  │  GET /range/B2C4E ─────────────►  │
  │                                    │
  │  ◄──────────────── 847 suffixes   │
  │  "EF245872B2A3C5040FE...:47832"   │  ← found! breached 47,832 times
  │  "A4D91F3C8B...:1"                │
  │  ...                               │
  │                                    │
  │  Local match → BREACHED            │
```

#### Why SHA-1 (Not SHA-256)?

HIBP uses SHA-1 specifically because the database was built before SHA-256 was the standard for this use case. SHA-1 is cryptographically broken for collision resistance, but for a lookup index it's irrelevant — you're not using it as a security primitive, you're using it as a hash key into their database.

## Part 2 — Implementation in PassGuard

#### `breach.js` — The New Shared Module

```javascript
// shared/breach.js

/**
 * Checks the password against HaveIBeenPwned Pwned Passwords via k-Anonymity.
 * @param {string} password
 * @returns {Promise<{ breached: boolean, count: number }>}
 */
export async function checkBreach(password) {
  const encoder = new TextEncoder();
  const data = encoder.encode(password);

  // SHA-1 via SubtleCrypto (WebCrypto API)
  const hashBuffer = await crypto.subtle.digest('SHA-1', data);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  const hashHex = hashArray.map(b => b.toString(16).padStart(2, '0')).join('').toUpperCase();

  const prefix = hashHex.slice(0, 5);
  const suffix = hashHex.slice(5);

  const response = await fetch(`https://api.pwnedpasswords.com/range/${prefix}`, {
    headers: { 'Add-Padding': 'true' }  // prevents traffic-analysis attacks via response size
  });

  if (!response.ok) {
    // Fail open — don't block the UI on API failure, just skip the check
    return { breached: false, count: 0, error: 'API unavailable' };
  }

  const text = await response.text();
  const lines = text.split('\r\n');

  for (const line of lines) {
    const [lineSuffix, countStr] = line.split(':');
    if (lineSuffix === suffix) {
      const count = parseInt(countStr, 10);
      return { breached: true, count };
    }
  }

  return { breached: false, count: 0 };
}
```

Key implementation details:

- **`crypto.subtle.digest('SHA-1', ...)`** — WebCrypto's `SubtleCrypto` interface. Async, no library needed, available in all modern browsers and in Chrome extension service workers.
- **`Add-Padding: true` header** — HIBP pads all range responses to the same length, preventing an attacker who can observe response packet sizes from narrowing down which prefix you queried.
- **Fail open on API error** — if the network is unavailable (user is offline, which PassGuard explicitly supports), the breach check returns `{ breached: false }` rather than throwing and breaking the whole analyzer. A network-unavailable state is logged in the UI as "Breach check unavailable — offline."

#### Wiring Into the Scorer

```javascript
// shared/scorer.js — updated computeScore signature
export function computeScore(strength, wordlist, patterns, username, personalResult, breachResult) {
  let score = /* ... existing weighted sum ... */;

  // Breach penalty — applied after all other scoring
  if (breachResult?.breached) {
    const count = breachResult.count;
    if      (count > 1_000_000) score -= 60;  // Critical — top breach lists
    else if (count > 100_000)   score -= 45;  // Very High
    else if (count > 10_000)    score -= 35;  // High
    else if (count > 1_000)     score -= 20;  // Medium
    else                        score -= 10;  // Low — appeared once, still notable
  }

  return Math.max(0, Math.min(100, Math.round(score)));
}
```

The breach penalty is **additive to the existing personal attack penalty** — both can apply simultaneously, and the combined floor is 0 (never negative).

#### UI Integration in `app.js`

```javascript
// app.js — debounced input handler
async function onPasswordInput(password) {
  const strength  = analyseStrength(password);
  const wordlist  = checkWordlist(password);
  const patterns  = detectPatterns(password);
  const username  = checkUsername(password, currentUser);
  const personal  = await runPersonalCheck(password);

  // Breach check: fire and forget — update UI when it resolves
  checkBreach(password).then(breachResult => {
    const finalScore = computeScore(strength, wordlist, patterns, username, personal, breachResult);
    updateBreachUI(breachResult);
    updateScoreUI(finalScore);
  });

  // Show intermediate score without breach result while API responds
  const interimScore = computeScore(strength, wordlist, patterns, username, personal, null);
  updateScoreUI(interimScore);
}
```

The "fire and forget then update" pattern means the UI responds instantly with the local score, then silently updates once the HIBP response comes back — typically in 80–200ms on a normal connection. The user never waits for a network request before seeing feedback.

## Part 3 — Extension Considerations

In the Chrome extension context, the breach check runs in the **background service worker** rather than the content script. Reasons:

1. Content scripts run in the context of the host page — making a `fetch()` from there would expose the request in the host page's network tab (DevTools → Network) if the page has any listener on service worker messages.
2. Background service workers make requests in their own isolated context, invisible to the host page's DevTools.
3. The service worker caches the HIBP response for the same 5-char prefix for 60 seconds — if the user types variations of the same password, the second lookup hits a local cache rather than re-fetching.

```javascript
// background.js — prefix cache
const prefixCache = new Map();  // prefix → { suffixes, timestamp }
const CACHE_TTL = 60_000;       // 60 seconds

async function cachedRangeQuery(prefix) {
  const cached = prefixCache.get(prefix);
  if (cached && Date.now() - cached.timestamp < CACHE_TTL) {
    return cached.suffixes;
  }
  const res = await fetch(`https://api.pwnedpasswords.com/range/${prefix}`, {
    headers: { 'Add-Padding': 'true' }
  });
  const suffixes = await res.text();
  prefixCache.set(prefix, { suffixes, timestamp: Date.now() });
  return suffixes;
}
```

## What I Practiced / Built

- Understood the k-Anonymity model for privacy-preserving breach database lookups
- Implemented `checkBreach()` in `shared/breach.js` using `crypto.subtle.digest('SHA-1', ...)` — no library, no backend
- Added breach penalty tiers to `scorer.js` (−10 to −60 pts based on breach count)
- Integrated the breach check as an async "fire and update" in `app.js` so the UI never blocks on network
- Added prefix-level LRU cache in the extension's background service worker to avoid redundant API calls
- Added `Add-Padding` header to prevent response-size traffic analysis

## Questions I Still Have

- Should the breach check be opt-in or opt-out in the extension? Some users may not want any external requests even with k-Anonymity.
- Is there a meaningful privacy difference between querying HIBP from the extension (background SW) vs. from the web app (direct tab)? Both reveal the same 5-char prefix to HIBP's CDN.
- HIBP's dataset is updated periodically — how should the extension surface staleness (e.g., "database last updated: 3 months ago")?
- Would it make sense to cache a local copy of the top 10,000 most-common hashes (prefix + suffix) to handle the worst offenders fully offline?
