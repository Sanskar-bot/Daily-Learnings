#### npm Workspaces тАФ How the Monorepo Is Wired

An npm workspace is a single repository where multiple packages share one `node_modules` folder. Instead of installing dependencies separately in `core/`, `backend/`, `extension/`, and `web/`, a single `npm install` at the root resolves everything тАФ shared packages are hoisted, duplicates are deduplicated.

The root `package.json` declares all four packages as workspaces:

```json
{
  "name": "vaultzero",
  "workspaces": ["core", "backend", "extension", "web"],
  "scripts": {
    "core:build":       "npm run build --workspace=core",
    "backend:dev":      "npm run dev --workspace=backend",
    "extension:dev":    "npm run dev --workspace=extension",
    "web:dev":          "npm run dev --workspace=web",
    "prisma:generate":  "npm run prisma:generate --workspace=backend",
    "prisma:migrate":   "npm run prisma:migrate --workspace=backend",
    "test":             "npm test --workspaces --if-present"
  }
}
```

#### TypeScript Configuration Strategy

Each package has its own `tsconfig.json`, but they all extend a common set of strict settings:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "outDir": "./dist",
    "declaration": true,
    "sourceMap": true
  }
}
```

`strict: true` enforces: `strictNullChecks`, `noImplicitAny`, `strictFunctionTypes`, `strictPropertyInitialization`. `noUncheckedIndexedAccess` adds `undefined` to every array/object index access тАФ catches off-by-one bugs at compile time.

#### The Prisma Schema тАФ 5 Models, Every Field Intentional

```prisma
model User {
  id              String   @id @default(uuid()) @db.Uuid
  email           String   @unique
  argon2Salt      String   @map("argon2_salt")       // per-user, stored as hex
  wrappedVaultKey String   @map("wrapped_vault_key") // AES-GCM(KEK, vaultKey) as base64
  recoveryHash    String?  @map("recovery_phrase_hash") // nullable тАФ optional feature
  createdAt       DateTime @default(now()) @map("created_at")
}
```

Three things worth understanding about this schema:

1. **`argon2Salt` is per-user** тАФ even two users with the same master password produce completely different KEKs, because their salts differ. The salt is 32 random bytes, generated once at registration, stored in plaintext (salts are public тАФ they're not secrets).

2. **`wrappedVaultKey` not `vaultKey`** тАФ the server never sees the raw vault key. It only stores the vault key *after* it has been encrypted by the user's KEK. The KEK is derived on-device and never transmitted.

3. **`RefreshToken.used` boolean** тАФ single-use token rotation. When a client uses a refresh token, the server marks it `used = true` and issues a new one. If the old token is ever presented again, it's treated as a replay attack тАФ all tokens for that user are immediately invalidated.

#### Security Rules Baked in from Day 1

These are non-negotiable and enforced by code review on every commit:

- тЭМ `Math.random()` тЖТ always `crypto.getRandomValues()`
- тЭМ `localStorage` for keys тЖТ extension service worker memory only
- тЭМ `console.log()` with passwords/keys тЖТ never log sensitive material
- тЭМ `innerHTML` with user content тЖТ XSS vector, never
- тЬЕ Fresh random IV on every AES-256-GCM call
- тЬЕ `try/catch` around every crypto operation
- тЬЕ TypeScript `strict: true` in every `tsconfig.json`
- тЬЕ Prisma parameterised queries only тАФ no raw SQL string concatenation
