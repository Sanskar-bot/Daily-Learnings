### PassGuard — Day 17: Firefox & Edge Compatibility (Cross-Browser MV3)

> **Context:** Day 17 of the PassGuard build. The extension runs flawlessly on Chrome. But Chrome's
> Manifest V3 and Firefox's WebExtensions API are *mostly* compatible — with a handful of
> critical gaps that silently break things. Today: port the extension to Firefox and Edge,
> document every incompatibility found, and ship a single codebase that runs on all three browsers
> without a build step or branching logic.

## Part 1 — MV3 vs Firefox's MV3 — The Incompatibility Map

Firefox shipped its own MV3 support in Firefox 109, but with intentional deviations from Chrome's spec. The gaps that actually affected PassGuard:

| Feature | Chrome MV3 | Firefox MV3 | PassGuard Impact |
|---|---|---|---|
| Service Worker lifetime | Event-driven, may be killed | Event-driven, may be killed | ✅ Same — state in storage |
| `chrome.*` namespace | `chrome.*` only | Both `browser.*` and `chrome.*` | ✅ Firefox polyfills `chrome.*` |
| `offscreen` API | ✅ Supported | ❌ Not supported | ⚠️ Not used in PassGuard |
| `action.setBadgeText()` | ✅ Works | ✅ Works | ✅ Compatible |
| `web_accessible_resources` format | MV3 object format | MV3 object format | ✅ Compatible |
| ES module workers (`type: 'module'`) | ✅ Supported | ❌ Not supported in workers | ❌ **Breaking** — workers fail silently |
| `chrome.storage.session` | ✅ Chrome 102+ | ❌ Not supported | ⚠️ Fallback needed |
| `declarativeNetRequest` | ✅ Supported | ✅ Firefox 113+ | ✅ Compatible |
| Persistent background script option | ❌ Removed | ✅ `"persistent": true` option | Not needed |

Two issues actually broke PassGuard on Firefox.

## Part 2 — Fix 1: ES Module Workers in Firefox

#### The Problem

Chrome supports `new Worker(url, { type: 'module' })` — the worker script can use `import` statements natively. Firefox **does not** support module workers; passing `{ type: 'module' }` causes the Worker to fail silently (no error, no output).

#### The Fix — Bundled Worker via `importScripts`

Instead of using ES module syntax inside the worker, the worker script was refactored to use `importScripts()` — the classic Worker API that all browsers support:

```javascript
// workers/analysis.worker.js — cross-browser version
// Replace: import { analyseStrength } from '../shared/strength.js';
// With:    importScripts via a self-contained bundle

importScripts(
  '../shared/strength.js',
  '../shared/patterns.js',
  '../shared/wordlist.js',
  '../shared/username.js',
  '../shared/personalDictionaryGenerator.js',
  '../shared/personalDictionaryScorer.js'
);

// Each imported file must export to globalThis instead of using export keyword
// OR — cleaner approach: detect context and conditionally export

self.onmessage = function(event) { /* ... same as before ... */ };
```

The cleaner approach: each shared module was updated to detect whether it's being loaded as a module or via `importScripts`:

```javascript
// shared/strength.js — cross-context export pattern
function analyseStrength(password) { /* ... */ }
function getCharsetSize(password)  { /* ... */ }

// Support both: ES module import AND importScripts() in classic workers
if (typeof module !== 'undefined') {
  // Node.js / ES module context
  module.exports = { analyseStrength, getCharsetSize };
} else if (typeof self !== 'undefined' && !self.window) {
  // Classic Web Worker context (importScripts)
  self.analyseStrength = analyseStrength;
  self.getCharsetSize  = getCharsetSize;
}
// ES module export (for import statements)
export { analyseStrength, getCharsetSize };
```

This dual-export pattern lets the same file work in: the web app (ES module import), Chrome's module worker, Firefox's classic worker (importScripts), and Node.js test scripts.

## Part 3 — Fix 2: `chrome.storage.session` Fallback

`chrome.storage.session` stores data that persists for the browser session but clears on browser restart. PassGuard used it to hold the in-memory unlock state (whether the master password has been entered this session).

Firefox doesn't support `storage.session`. The fix: a thin wrapper that falls back to `storage.local` with a session-token key:

```javascript
// shared/sessionStore.js
const SESSION_KEY = '__passguard_session_token';

export const sessionStore = {
  async get(key) {
    if (chrome.storage.session) {
      return chrome.storage.session.get(key);
    }
    // Firefox fallback — use local storage with a session-scoped prefix
    const result = await chrome.storage.local.get(`session_${key}`);
    return { [key]: result[`session_${key}`] };
  },

  async set(items) {
    if (chrome.storage.session) {
      return chrome.storage.session.set(items);
    }
    const prefixed = Object.fromEntries(
      Object.entries(items).map(([k, v]) => [`session_${k}`, v])
    );
    return chrome.storage.local.set(prefixed);
  },

  async clear() {
    if (chrome.storage.session) {
      return chrome.storage.session.clear();
    }
    // Clear all session_ prefixed keys from local storage
    const all = await chrome.storage.local.get(null);
    const sessionKeys = Object.keys(all).filter(k => k.startsWith('session_'));
    return chrome.storage.local.remove(sessionKeys);
  }
};
```

The trade-off: on Firefox, "session" data technically persists through a browser restart (it's in `local` storage). PassGuard addresses this by storing a session timestamp alongside the unlock state and treating any state older than 8 hours as expired — effectively simulating session behavior.

## Part 4 — Edge Compatibility

Edge uses Chromium's engine and supports Chrome's exact MV3 spec. The only Edge-specific change: Edge's extension store requires a `"edge_action"` key in certain older Edge builds. Since PassGuard targets the modern Edge Chromium (not legacy EdgeHTML), this wasn't needed. All Chrome fixes apply to Edge automatically.

The extension was sideloaded on Edge via `edge://extensions → Load unpacked`. All features — content script injection, Shadow DOM widget, popup dashboard, background badge — worked identically to Chrome.

## Part 5 — Browser Detection Utility

Rather than scattering `typeof chrome.storage.session !== 'undefined'` checks everywhere, a central `browserCompat.js` module was added:

```javascript
// shared/browserCompat.js
export const browserCompat = {
  isFirefox: navigator.userAgent.includes('Firefox'),
  isEdge:    navigator.userAgent.includes('Edg/'),
  isChrome:  !navigator.userAgent.includes('Firefox') &&
             !navigator.userAgent.includes('Edg/'),

  supportsStorageSession: !!chrome?.storage?.session,
  supportsModuleWorkers:  !navigator.userAgent.includes('Firefox'),
};
```

All feature-check branches in the codebase now read from `browserCompat` instead of inline `typeof` checks — easier to test, easier to mock in unit tests.
