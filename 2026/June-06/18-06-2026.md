## Part 1 — The Math: Why Passphrases Win

#### Random Character Passwords

A 10-character password from the full printable ASCII set (95 characters):

```
Entropy = length × log₂(charset)
        = 10 × log₂(95)
        = 10 × 6.57
        = 65.7 bits
```

That's a good password. But it looks like `nK7@mQ!2Lp` — nearly impossible to memorize.

#### Diceware Passphrases

The EFF Large Wordlist has **7,776 words** (6 dice rolls per word: 6⁵ = 7,776). Each word contributes:

```
log₂(7,776) = 12.92 bits per word
```

A 5-word passphrase:

```
5 × 12.92 = 64.6 bits
```

Almost identical entropy to a 10-character complex password — but the passphrase `coral freight lamp sunrise grid` is:
- Memorable (real English words)
- Typeable without a symbol keyboard
- Speakable over a phone without ambiguity
- Signable into a hardware key at arm's length

A 6-word passphrase gives **77.5 bits** — stronger than most complex passwords users actually create.

#### The Key Insight

Entropy comes from **the size of the space the attacker must search**, not from visual complexity. An attacker cracking `coral-freight-lamp-sunrise-grid` must search a space of 7,776⁵ = ~2.8 × 10²³ combinations. An attacker cracking `Password1!` only needs to search a trivially small subset of the real human-chosen password space.

## Part 2 — The EFF Wordlist

The EFF (Electronic Frontier Foundation) published a Diceware wordlist specifically designed for passphrase use:

- **7,776 words** — exactly 6⁵, so any number of dice rolls maps to a word without remainder bias
- **No ambiguous words** — avoids words that sound like other words over the phone (`bare` vs `bear`)
- **No offensive words** — safe for any context
- **Variable length** — words range from 3–9 characters; shorter words keep passphrases typeable

PassGuard uses a curated subset of this list (1,000 common, unambiguous words) embedded directly in `data/wordlist_diceware.js` — no fetch required, fully offline.

## Part 3 — Implementation: `passphraseGenerator.js`

```javascript
// shared/passphraseGenerator.js
import { DICEWARE_WORDS } from '../data/wordlist_diceware.js';

/**
 * Generates a cryptographically secure Diceware passphrase.
 * @param {object} options
 * @param {number} options.wordCount     - Number of words (default: 5)
 * @param {string} options.separator     - Word separator (default: '-')
 * @param {boolean} options.capitalize   - Capitalize first letter of each word
 * @param {boolean} options.includeNumber - Append a secure random number
 * @param {boolean} options.includeSymbol - Append a secure random symbol
 * @returns {{ passphrase: string, entropy: number, words: string[] }}
 */
export function generatePassphrase(options = {}) {
  const {
    wordCount     = 5,
    separator     = '-',
    capitalize    = false,
    includeNumber = false,
    includeSymbol = false,
  } = options;

  const words = [];
  for (let i = 0; i < wordCount; i++) {
    const idx = secureRandomIndex(DICEWARE_WORDS.length);
    const word = DICEWARE_WORDS[idx];
    words.push(capitalize ? word[0].toUpperCase() + word.slice(1) : word);
  }

  let passphrase = words.join(separator);

  if (includeNumber) {
    passphrase += separator + secureRandomIndex(100).toString().padStart(2, '0');
  }

  if (includeSymbol) {
    const symbols = '!@#$%&*?';
    passphrase += symbols[secureRandomIndex(symbols.length)];
  }

  // Entropy calculation
  const wordEntropy   = wordCount * Math.log2(DICEWARE_WORDS.length);
  const numberEntropy = includeNumber ? Math.log2(100) : 0;
  const symbolEntropy = includeSymbol ? Math.log2(8) : 0;
  const totalEntropy  = wordEntropy + numberEntropy + symbolEntropy;

  return { passphrase, entropy: totalEntropy, words };
}

/**
 * Rejection-sampling secure random index — no modulo bias.
 */
function secureRandomIndex(max) {
  const limit = 0x100000000 - (0x100000000 % max);
  let val;
  do {
    val = crypto.getRandomValues(new Uint32Array(1))[0];
  } while (val >= limit);
  return val % max;
}
```

The rejection sampling is the same function used in the base generator (Day 9) — factored into a shared `secureRandom.js` utility to avoid duplication.

## Part 4 — Entropy Display in the UI

The passphrase generator needed a slightly different UI treatment than the character-based generator. For character passwords, the strength score (0–100) is the primary metric. For passphrases, **entropy in bits** is more meaningful — a user who understands "77 bits" makes better decisions than one who sees a "Strong" label.

The UI now shows both:

```
┌───────────────────────────────────────────────────┐
│  coral-freight-lamp-9-sunrise                      │
│                                                    │
│  Entropy: 71.2 bits  ·  Estimated crack time:      │
│  GPU cluster: 3.6 million years                    │
│                                                    │
│  Score: 96 / 100 — Very Strong                     │
│  [Regenerate]   [Copy]                             │
└───────────────────────────────────────────────────┘
```

`estimateCrackTimes()` from `bruteforce.js` (Day 7) already accepts entropy in bits directly — the passphrase generator feeds `totalEntropy` into it without modification.

## Part 5 — Integration with the Scorer

Running a Diceware passphrase through the existing `checkWordlist()` Trie would falsely penalize it — every word is in the dictionary, that's the point. The scorer was updated to detect passphrase mode:

```javascript
// shared/scorer.js — passphrase mode
export function computeScore(strength, wordlist, patterns, username, personal, breach, isPassphrase = false) {
  if (isPassphrase) {
    // Passphrases are scored purely on entropy — no wordlist penalty
    // (dictionary words are intentional; the randomness comes from selection, not obscurity)
    const entropyScore = Math.min(100, Math.floor(strength.entropy / 1.2));
    const personalPenalty = personal?.rank <= 1000 ? -20 : 0;
    const breachPenalty   = breach?.breached ? -30 : 0;
    return Math.max(0, Math.min(100, entropyScore + personalPenalty + breachPenalty));
  }
  // ... existing non-passphrase scoring ...
}
```

The core insight: for passphrases, the wordlist match is not a weakness — it's the design. Penalizing it would incorrectly score a 5-word Diceware passphrase as "Weak" because all five words appear in the dictionary.

## Questions I Still Have

- Should the separator be randomized per-word instead of consistent? (`coral-freight=lamp#sunrise`) This adds ~3 bits of entropy but makes the passphrase much harder to type consistently.
- How does the HIBP breach check interact with passphrases — is any Diceware combination likely to appear in breach databases? (Almost certainly no — the combination space is too large.)
- Should the wordlist be user-selectable (EFF, Beale, custom)? Or is one curated list better for consistency?
- Does capitalization add meaningful entropy if the attacker knows the target uses PassGuard's passphrase generator? (No — it adds exactly `log₂(2^wordCount)` = `wordCount` bits, trivially small.)
