## Part 2 — What Worked Well

#### 1. Zero-Dependency Constraint

Locking "no libraries, no build step" on Day 1 was the single best architectural decision. It forced me to:
- Learn `SubtleCrypto` instead of reaching for a crypto library
- Learn the Trie data structure instead of using a fuzzy-matching package
- Learn `MutationObserver` instead of reaching for a framework's reactive system
- Learn Web Workers instead of reaching for a concurrency library

Every "hard" part of the project became a learning opportunity because the easy shortcuts were off the table.

#### 2. Forward-Compat API Design (Scorer)

Designing the scorer's function signature to accept an optional `personalResult` on Day 6 — before the personal attack engine existed — meant zero API changes were needed on Day 13. The contract was right from the start. The same decision made adding `breach` and `isPassphrase` params on Days 15 and 18 painless.

#### 3. Shared Engine Modules

Making `shared/` the single source of truth for all analysis logic, imported identically by the web app and the extension, meant every bug fix and feature addition applied everywhere simultaneously. There was never a "web app version" and an "extension version" of the scorer drifting apart.

## Part 3 — What I'd Do Differently

#### 1. Plan the Worker Architecture on Day 1

The Web Worker refactor (Day 16) was harder than it needed to be because the worker wasn't planned from the start. If I'd designed `analysis.worker.js` before writing `app.js`, the message-passing interface would have been the primary API instead of a retrofit. The stale-ID discard pattern and the dual-export module compatibility work were both symptoms of a worker being added after the fact.

**Lesson:** For any feature that involves user-visible computation (debounced analysis, generative algorithms), decide on Day 1 whether it belongs on the main thread or a worker, and design the interface accordingly.

#### 2. Cross-Browser Testing Earlier

Finding the Firefox module-worker incompatibility on Day 17 (after 16 days of Chrome-only development) required refactoring the dual-export pattern across all shared modules. If I'd tested on Firefox on Day 10 (when the extension was first scaffolded), the dual-export pattern would have been baked in from the start rather than bolted on.

**Lesson:** Test on target browsers from the first day the extension runs, not the last.

#### 3. The Sequential-Pattern False Positive (Day 3 → Day 14)

Flagging this bug on Day 3 and living with it until Day 14 was a mistake in prioritization. False positives in pattern detection erode user trust — if the scorer penalizes a legitimately strong password for a "keyboard walk" that isn't one, the user stops trusting the tool. Bugs that affect scoring accuracy should be fixed immediately, not deferred.

## Part 4 — What's Next (PassGuard v2)

The logical next features, in priority order:

#### Tier 1 — High Value, Feasible Immediately

1. **Password history & reuse detection across sites** — extend the SHA-256 hash storage (Day 13) to track which domain each password was used for, and warn when the same hash is reused on a new site.
2. **Export vault** — AES-256-GCM encrypted JSON export, decryptable only with the master password. Needed before PassGuard can be a practical daily-driver.
3. **Auto-fill** — content script detects `<input type="password">` at login forms and injects a "Fill with PassGuard" button. Requires form-field detection heuristics beyond what's currently in the widget.

#### Tier 2 — Significant Engineering

4. **Sync across devices** — WebCrypto key derivation from the master password, encrypt the vault client-side, sync the ciphertext via a minimal backend (or use Chrome's `storage.sync` for small vaults). Biggest architectural addition — needs a backend or a sync service.
5. **TOTP (2FA code generator)** — implement RFC 6238 TOTP in pure JS (WebCrypto HMAC-SHA1) and store TOTP secrets alongside credentials. Turns PassGuard into a combined password + 2FA manager.
6. **Accessibility audit** — full WCAG 2.1 AA compliance pass on the popup and widget: keyboard navigation, screen reader labels, focus management in the Shadow DOM.

#### Tier 3 — Research / Experimental

7. **Markov chain attack simulation** — instead of combinatorial personal dict generation, use a character-level Markov model trained on the user's profile strings to generate statistically realistic guesses. More accurate attack simulation, significantly more complex.
8. **Offline breach cache** — pre-bundle the SHA-1 prefix+suffix for the top 100,000 most-breached passwords. Eliminates the HIBP API dependency for the most critical cases, enabling fully offline breach detection.

## Part 5 — Project Self-Assessment

Looking back at the original problem statement — *"traditional password rules catch almost nothing real attackers exploit"* — PassGuard delivers on that premise:

- **Entropy math** catches `Password123!` (high character variety, terrible entropy) that all rule-based validators pass.
- **Trie substring matching** catches `myDr4g0nPassword` that exact wordlist checks miss.
- **Personalized attack simulation** catches `Sanskar2005` that no generic tool would flag.
- **HIBP integration** catches any password that's actually appeared in a real breach, regardless of its mathematical strength.
- **Zero backend, fully offline** — the original non-negotiable constraint — held for 20 days and across every feature addition.

The project proved that a world-class security tool is achievable with vanilla JavaScript, the WebCrypto API, and a well-designed data structure. No framework required.

## Questions I Still Have

- At what point does a "zero backend" constraint become a real product limitation? Sync across devices and encrypted cloud backup both need a server. Is the right answer "build the minimal backend" or "integrate with an existing sync service (Notion, GitHub Gist, Dropbox)"?
- Is the TOTP secret storage security model acceptable? TOTP secrets stored in `chrome.storage.local` are encrypted by the OS keychain on most systems — but this is OS-level, not application-level, encryption.
- Should PassGuard remain a personal/portfolio project or be submitted to the Chrome Web Store? The Web Store requires a privacy policy, review by Google, and ongoing maintenance. What's the threshold for that commitment?
