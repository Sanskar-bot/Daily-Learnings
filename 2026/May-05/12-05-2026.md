### E2E Encryption Attack Lab — Act 2 Complete: MITM Key Swap

> **Project Context:** Day 3. Today's goal was to build the actual attacker — a MITM proxy that
> silently intercepts and decrypts the "secure" E2E conversation by swapping RSA public keys.
> This is the core attack that proves encryption without verification is insufficient.

#### How the MITM Key Swap Attack Works
- The MITM sits between Alice and Bob, acting as a transparent TCP proxy
- When Alice sends her RSA public key, MITM **intercepts it and swaps it with its own public key** before forwarding to Bob
- Same in reverse — Bob's public key is swapped with MITM's key before reaching Alice
- Alice and Bob each believe they have the other's real key — they actually have the MITM's key
- When Alice encrypts the AES session key with "Bob's" key (actually MITM's), MITM can decrypt it
- MITM then re-encrypts the session key with Bob's *real* public key and forwards it
- Both sides complete the handshake believing it was secure — MITM now holds the AES session key
- Every subsequent AES-GCM message is decrypted, logged, and re-encrypted by the MITM

#### `mitm.py` — TCP Proxy with Key Swap
- Proxy listens on **port 8080**, connects upstream to the relay server on **port 9000**
- Generates its own RSA-2048 keypair at startup (the "evil" key pair)
- Maintains session state to track Alice's and Bob's connections separately
- Implements full key swap and session key interception logic

#### `mitm_addon.py` — mitmproxy Integration
- Wrote a **mitmproxy addon script** that integrates the key swap logic into the professional `mitmproxy` tool
- Learned how mitmproxy's addon API works — `request()`, `response()`, and `tcp_message()` hooks
- Understood how professional security tools extend their functionality through scripting

#### Challenge Solved — Cross-Thread State Sharing (Session Pairing)
- **Problem:** The MITM spawns separate threads for Alice's and Bob's connections. When Alice's `session_key` message arrived, Bob's real public key wasn't in the state dict yet — causing a `KeyError` / `None` decryption failure.
- **Root cause:** Thread scheduling is non-deterministic; Alice's thread was processing faster than Bob's.
- **Fix:** Used a `threading.Event` object — Alice's session_key handler **blocks** (`event.wait()`) until Bob's `key_exchange` handler sets the event, confirming Bob's real public key has been stored. This creates FIFO session pairing via a shared state dictionary.
- **Lesson:** `threading.Event` is the right tool for cross-thread signalling when one thread must wait for another to complete a specific action.
