## What I Learned

### E2E Encryption Attack Lab — Act 3: Fingerprint Verification Defeats MITM

> **Project Context:** Day 4. Today's goal was to implement the *defense* — showing how out-of-band
> key fingerprint verification completely defeats the MITM key swap attack. This mirrors exactly
> how WhatsApp Safety Numbers and Signal's safety numbers work in production.

#### Key Fingerprint Verification — The Concept
- After the RSA key exchange, each client computes the **SHA-256 hash (fingerprint)** of the public key they *received*
- Both parties display their fingerprint and compare them **out-of-band** (verbally, in person, or via a trusted channel)
- If a MITM swapped keys, Alice has MITM's key and Bob has MITM's key → their fingerprints will **both differ** from the real party's key fingerprint
- If no MITM, both fingerprints are identical → safe to proceed
- This is the **Trust On First Use (TOFU)** model's verification step — the same model used by SSH

#### `verified` Mode Implementation
- Added `--mode verified` flag to both `alice.py` and `bob.py`
- After key exchange: both clients display **two fingerprints** side by side:
  - `My key fingerprint:` — fingerprint of their own public key (what the other person should see)
  - `Received key fingerprint:` — fingerprint of the key they received (what they're actually using)
- User is prompted: *"Does this match what the other person sees? [yes/no]"*
- On `no` → prints `⚠ MITM ATTACK DETECTED — Aborting session.` → socket closed → program exits
- On `yes` → chat proceeds with full confidence in the key

#### Real-World Parallels Learned
- **WhatsApp Safety Numbers** — exact same mechanism: compare 60-digit codes out-of-band
- **Signal Safety Numbers** — same, with optional QR code scanning for in-person verification
- **SSH host key verification** — first connection shows fingerprint; subsequent connections warn if it changes
- **TLS Certificate Authorities** — solve the same problem at scale via trusted third-party signing

#### Challenge Solved — Displaying Wrong Fingerprint
- **Problem:** Alice was printing the fingerprint of *her own* key (the one she sent), not the key she received from Bob. This made it impossible to visually detect the swap — both sides appeared to show the same value.
- **Root cause:** Code was calling `fingerprint(my_public_key)` instead of `fingerprint(received_public_key)`.
- **Fix:** Clarified the display to explicitly label and separately compute:
  - `fingerprint(my_keypair.public_key())` → "My key" column
  - `fingerprint(received_key)` → "Received key" column
  - Showing them side by side in a formatted verification box made the mismatch immediately obvious when MITM was active.
