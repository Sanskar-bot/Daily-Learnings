## What I Learned

### E2E Encryption Attack Lab — Act 2 Base: RSA Key Exchange + AES-GCM Encryption

> **Project Context:** Day 2 of the MITM Lab. Today's goal was to add real encryption to Alice and Bob
> so that Act 2 can demonstrate that encryption *alone* (without key verification) is not enough.

#### RSA Key Exchange Protocol
- Implemented the full **public key exchange handshake**:
  1. Alice generates an RSA-2048 keypair
  2. Both Alice and Bob send their public keys to each other via the relay server
  3. Alice generates a random 32-byte AES session key
  4. Alice encrypts the session key with Bob's RSA public key and sends it
  5. Bob decrypts the session key using his RSA private key
  6. Both sides now share the same AES-256-GCM session key without ever sending it in plaintext
- Understood this is the fundamental pattern behind TLS and other secure protocols

#### AES-256-GCM Session Encryption
- After the handshake, all messages are encrypted using **AES-256-GCM** with the shared session key
- Each message generates a fresh random **96-bit nonce** (IV) — critical for GCM security
- The nonce is prepended to the ciphertext and sent alongside it
- **Authentication tag** (16 bytes) embedded in GCM output ensures message integrity — any tampering fails decryption

#### Message Protocol Design
- Defined a simple JSON message envelope: `{"type": "key_exchange" | "session_key" | "message", "data": "..."}`
- This type field is critical — it lets clients and the MITM proxy distinguish between handshake and chat messages

#### Wireshark — Ciphertext Verification
- Re-captured Wireshark traffic with encryption enabled
- ASCII pane now shows **gibberish** — confirming AES-GCM encryption is working
- This sets up the Act 2 narrative: encryption exists, but is it trustworthy without key verification?

#### Challenge Solved — Race Condition in Key Exchange
- **Problem:** Alice was sending the AES session key before Bob's RSA public key had arrived at the server — causing a `null key` decryption error on Bob's side.
- **Root cause:** Both clients connected and immediately started sending without waiting for the other's key.
- **Fix:** Added a **blocking receive loop** in Alice that waits specifically for a message of type `key_exchange` before generating and sending the AES session key. This ensures Bob's public key is received before Alice proceeds.
- **Lesson:** Distributed systems require explicit synchronization — never assume message ordering.
