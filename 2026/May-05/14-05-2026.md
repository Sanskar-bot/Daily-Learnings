#### Threading Model — Flask + TCP Proxy Coexistence
- Understood the problem: Flask's `app.run()` is **blocking** — it takes over the main thread
- A naïve implementation would prevent the TCP proxy from ever running
- Learned the daemon thread pattern as the standard solution for this scenario

#### Challenge Solved — Flask Blocking the TCP Proxy
- **Problem:** Starting Flask with `app.run()` blocked the main thread, preventing the TCP proxy loop from executing. The MITM stopped working the moment the dashboard was added.
- **Fix:** Started Flask in a **daemon thread**:
  ```python
  threading.Thread(target=app.run, kwargs={"port": 7000}, daemon=True).start()
  ```
  Setting `daemon=True` means the Flask thread will automatically die when the main program exits — no cleanup needed. The main thread continues running the TCP proxy loop uninterrupted.
- **Lesson:** Daemon threads are ideal for background services (APIs, loggers, health checks) that should live and die with the main process.

#### UI/UX Concepts Applied
- Polling interval trade-off: 1 second is responsive enough for a demo without hammering the server
- Color semantics in security dashboards — red/amber/green maps to threat severity
- Keeping the dashboard stateless (server holds all state, client just renders) simplifies the architecture
