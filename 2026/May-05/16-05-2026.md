#### What "Clean Run" Means (and Why It Matters)
- A "clean run" means starting from fresh terminals with **no leftover state** — no cached keys, no stale sockets, no previously started servers
- This is critical for a demo: if it only works on *your* machine after a warm-up, it's not reliable
- Learned to test the "unboxing experience" — what someone sees when they clone the repo and run it for the first time

#### Code Quality — Docstrings and Inline Comments
- Added **module-level docstrings** to all `.py` files explaining the file's role in the system
- Added **function-level docstrings** following the format: purpose, parameters, return value
- Added **inline comments** at non-obvious logic points — especially around the threading synchronization, key swap logic, and AES nonce handling
- Understood the difference: docstrings explain *what* and *why*; inline comments explain *how* when the code isn't self-evident
